# 專案架構 (Project Architecture)

本文件描述了 `ok_multipl_poker` 專案的軟體架構、模組職責以及組件間的依賴關係。

## 架構概覽

本專案遵循分層架構設計，並特別針對 Firebase 即時同步進行優化。主要依賴原則為 **UI → Controller (Provider) → Entity/Internals**。

## 核心層級職責

### 1. 展示層 (Presentation Layer)
*   **職責**: 處理 UI 構建、遊戲桌面渲染、動畫效果以及使用者操作輸入。
*   **Play Session**: 不同遊戲的獨立桌面實作，負責將數據狀態轉化為卡牌佈局。

### 2. 業務控制層 (Control Layer)
*   **職責**: 封裝與 Firebase 的交互邏輯、處理加入/離開房間、出牌請求提交以及監聽遠端狀態更新。
*   **Controllers**: 包含 `FirestoreRoomController` 與特定遊戲的控制邏輯。

### 3. 核心數據與規則層 (Core & Data Layer)
*   **職責**: 負責定義資料模型與純粹的遊戲邏輯（牌型判斷、勝負決定）。
*   **Entities**: 對應 Firestore 文件的 POJO，包含 JSON 序列化邏輯。
*   **Game Internals**: 封裝撲克牌組、洗牌、出牌規則等與平台無關的純 Dart 邏輯。

## 依賴規則
*   **核心解耦**: `game_internals` 不應依賴於 Firebase 或 UI 框架，應為純粹的業務規則。
*   **單向流**: UI 發起 Actions -> Controller 寫入 Firebase -> Controller 監聽 Firebase 更新 -> UI 重新渲染。
